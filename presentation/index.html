The Problem

What drove this proof of concept
01
Shared Tablets, Multiple Drivers
Multiple drivers share the same tablets across shifts and depots. Any enrolled driver must be able to log in on any tablet — credentials cannot be device-specific.
02
Field Workers Need Speed
Drivers work in the field, often with gloves or in harsh conditions. Authentication must complete in under 10 seconds. Passwords are impractical. Forgotten credentials block the shift.
03
Identity Verification Is Required — Not Just Authentication
A PIN alone is insufficient. Buddy-punching (one driver clocking in for another) must be prevented. The system must confirm that the person entering the PIN is the registered driver — not just someone who knows the code.
04
No Hardware Budget for Per-Driver Devices
Issuing one tablet per driver is not economical at scale. The solution must work on shared hardware without any per-tablet enrollment or configuration.

Why Not Passkey (WebAuthn)?

Strong technology — wrong fit for a shared-tablet fleet
✓  What Passkeys do well
  • Phishing-resistant — private key never leaves the device; no password to steal
  • No server-side secret — server only stores a public key, breach has no usable credential
  • Biometric UX — one-touch sign-in with Face ID / fingerprint on personal devices
  • Industry standard — FIDO2 / W3C WebAuthn, supported by all major browsers & OS
✗  Why it breaks for our fleet
  • Shared tablets — passkeys are tied to one device + one OS user account; 10 drivers sharing Tablet A is not supported
  • Sync requires one Google account per driver — passkeys sync via Google Password Manager only if tablets share the same account; a fleet of shared tablets with multiple drivers has no clean account model
  • No fleet admin model — no built-in way for ops to enroll, revoke, or reassign credentials across a device pool
  • Wrong problem domain — designed for web login on personal devices, not identity switching on shared hardware
Verdict: Passkeys are the right answer for personal-device consumer apps. For a fleet of shared tablets with multiple drivers, the device-bound model is a fundamental mismatch — not a configuration problem.

Solution A — NFC Card + PIN

Branch: feature/nfc-auth  ·  Fully implemented
Physical 2nd Factor Something You Have + Know
1
Admin Issues Card
Admin writes driver ID to NFC card via AdminScreen. Uses NDEF Text Record format — human-readable, standard.
›
2
Driver Taps Card
Driver holds NFC card to tablet. App reads driver ID from NDEF record instantly. No typing needed.
›
3
PIN Entry
Driver enters their personal PIN on the tablet numpad. Hash sent to cloud API for verification.
›
4
Access Granted
On PIN match, driver is routed to the Dashboard. Session start time is logged.
Tech stack:   Android NFC API  ·  NDEF Text Records  ·  Neon PostgreSQL  ·  Vercel Serverless  ·  OkHttp3
What Makes It Two-Factor
Card (something you have) + PIN (something you know). Both are required — losing either factor alone is not enough to log in.
Admin Enrollment
Admin writes driver ID to blank NFC card. If a card is lost, admin issues a replacement. No driver can enroll their own card without admin access.

Solution A — Strengths & Limitations

✓  Strengths
⚡ Fast UX — card tap takes under 1 second
🪪 True physical 2nd factor — something you have + something you know
👷 Familiar to drivers — similar to access-card systems they already use
📱 No biometric hardware required — works on any NFC-enabled tablet
🔧 Low implementation complexity — NDEF read + REST API call
✅ Card is tangible proof of enrollment — easy for ops to manage
✗  Limitations & Risks
🃏 Card can be lost, stolen, or cloned — no cryptographic binding
👁️ No liveness check — anyone holding the card can attempt PIN entry
🖨️ Requires card printing, distribution & re-issuance workflow at scale
🚫 No revocation mechanism — lost card requires direct DB intervention
📡 NFC reader availability not guaranteed on all tablets in the fleet
⚠️
Critical Gap: NFC cards store only a driver ID — no secret. A cloned card + guessed PIN grants full access. Buddy-punching is still possible if a driver shares their PIN with a colleague.

The Two ML Models Used

Two specialised models — each does one job only
🔍
Google
ML Kit
FREE · ON-DEVICE
Face Detection — Is there a real live face here?
A pre-trained Google model built into the app. It processes every camera frame and answers simple yes/no questions about the face it sees. We use its output to run the liveness challenge — it never decides who the person is.
👁 Are eyes open? → 0.0–1.0
😊 Is smiling? → 0.0–1.0
↔ Head turned how many degrees? → angle°
🧠
FaceNet
TFLite
OPEN SOURCE
Face Recognition — Who is this person?
An open-source neural network (Google Research, 2015) converted to run on Android. It takes a cropped face photo and converts it into 512 numbers that uniquely describe that face — like a fingerprint, but mathematical. The same person always produces numbers that are very close. Different people produce numbers that are far apart. Only those 512 numbers leave the device — never the photo.
📥 Input: 160×160px face crop
📤 Output: 512 numbers (embedding)
⚡ Runs fully on-device — no internet needed

Solution B — Authentication Flow

Branch: feature/facenet-auth  ·  Recommended
★ Recommended Something You Know + Something You Are No Special Hardware
1
Driver Picker
Driver selects their name from a list fetched from the cloud. Enrolled drivers shown in teal; non-enrolled shown dimmed and blocked from login.
2
PIN Entry
Driver enters 6-digit PIN. Server verifies using Argon2id (64 MB memory, 3 iterations) — the most resistant modern hashing algorithm. PIN never stored in plain text anywhere.
3
Liveness Challenge
ML Kit detects face. Two random challenges assigned from: BLINK · SMILE · LOOK LEFT · LOOK RIGHT. Each session gets a different random sequence — replay attacks fail.
4
Face Embedding (On-Device)
FaceNet TFLite model runs locally — 512-float L2-normalised vector produced from the face region. No face image ever leaves the device.
5
Cloud Match
Server decrypts stored embeddings (AES-256-GCM), computes cosine similarity against all 5 enrollment captures. Score ≥ 0.75 → granted. Fail 5× → 60-minute lockout.
Tech:   FaceNet TFLite  ·  ML Kit Face Detection  ·  CameraX  ·  Argon2id  ·  AES-256-GCM  ·  Neon PostgreSQL  ·  Vercel

Security Architecture — Solution B

Layered defence at every point in the authentication pipeline
PIN Protection
  • Argon2id hashing — 64 MB memory cost, 3 iterations, per-driver random salt
  • Takes ~300ms to verify — slows brute-force to ~3 attempts/minute
  • Raw PIN sent over HTTPS only — never stored anywhere in plain text
  • Pepper value in Vercel env — not in the database, rotatable independently
Biometric Privacy
  • Face images never leave the device — only 512 math floats transmitted
  • Embeddings encrypted at rest with AES-256-GCM before writing to DB
  • Encryption key (FLEET_EMBED_KEY) stored only in Vercel env variables
  • No raw biometric data stored or transmitted at any stage
Brute-Force Defence
  • Tablet lockout: 5 wrong PINs → 30-minute lock (local, timestamp-based)
  • Driver lockout: 5 wrong face attempts → 60-minute per-driver lock
  • Duplicate registration blocked: first enrollment scan checked against all stored drivers
  • Lockout requires no server round-trip — works offline
Liveness & Anti-Replay
  • Randomized 2-step liveness — different challenge pair every session
  • Neutral-then-action pattern prevents false triggers from still images
  • Only ML Kit classification values used — no camera frame transmitted
  • LOOK_LEFT + LOOK_RIGHT never paired — prevents confusion attacks
API Auth: All fleet endpoints require Bearer token (SYNC_SECRET). Absent or wrong token → 401 immediately.
Transport: HTTPS only. Certificate pinning recommended for production (not yet implemented).

Solution B — How It Helps & Where It Falls Short

How It Helps
  • ✓ Eliminates buddy-punching — face cannot be shared or delegated
  • ✓ Works on any tablet with a front camera — zero hardware upgrades
  • ✓ Any driver can log in on any tablet — cloud embedding lookup
  • ✓ Driver self-registration — no admin needed per enrollment
  • ✓ PIN hashed with Argon2id — strongest password algorithm available
  • ✓ Face embeddings encrypted at rest — GDPR-friendly biometric storage
  • ✓ Duplicate face check prevents one person creating multiple accounts
  • ✓ Randomized liveness defeats photo, video, and screen-replay attacks
Current Limitations
  • ✗ Lighting sensitive — poor light degrades face match score
  • ✗ Face match sensitivity not calibrated yet — the app decides "is this the right person?" by comparing a similarity score against a fixed value of 0.75. Set too high → real drivers get rejected. Set too low → a different person might get in. The right value can only be found by testing on actual driver faces
  • ✗ First scan has ~300ms delay while TFLite model loads
  • ✗ No admin dashboard yet — if a driver is locked out after 5 failed attempts or leaves the company, ops currently need direct database access to unlock or remove them
  • ✗ Enrollment takes ~45 seconds (5 face captures per driver)
  • ✗ HTTPS certificate pinning not yet implemented — without it, an attacker on the same network could intercept the face embedding or PIN using a fake certificate (man-in-the-middle attack)
